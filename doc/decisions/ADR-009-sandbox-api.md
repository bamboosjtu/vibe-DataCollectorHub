# ADR-009: Sandbox API 下游应用接口

## 状态

- 状态: 已接受
- 日期: 2026-05-06

## 背景

Data Collector Hub 作为数据枢纽，需要为多个下游应用提供数据服务。数字沙盘是其中一个下游应用，需要消费归一化后的数据。关键约束：

1. 下游应用不应该直接访问原始数据（raw_events）或上游特定字段
2. 不同下游应用可能需要不同的数据视图和接口语义
3. 系统需要保持开放，支持未来新增下游应用

## 决策

**为下游应用提供专用的 Sandbox API，返回标准化 DTO，不暴露原始 DCP 字段。**

```
┌─────────────────────────────────────────────────────────────┐
│  Data Collector Hub                                         │
│                                                             │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐     │
│  │  raw_events │───▶│ Normalizer  │───▶│ canonical_  │     │
│  │             │    │             │    │  entities   │     │
│  └─────────────┘    └─────────────┘    └──────┬──────┘     │
│                                                │            │
│                                         ┌──────┴──────┐     │
│                                         │ Sandbox API │     │
│                                         │             │     │
│                                         │ /dates      │     │
│                                         │ /map/       │     │
│                                         │   skeleton  │     │
│                                         │ /map/       │     │
│                                         │   summary   │     │
│                                         └──────┬──────┘     │
│                                                │            │
└────────────────────────────────────────────────┼────────────┘
                                                 │
                                                 ▼
                                        ┌─────────────────┐
                                        │  下游应用         │
                                        │  （如数字沙盘）    │
                                        └─────────────────┘
```

### Sandbox API 设计原则

1. **数据隔离**：返回 DTO，不暴露原始字段
2. **专用语义**：接口命名贴合下游应用需求
3. **标准化数据**：基于 canonical_entities，面向下游消费

### 当前接口

```
GET /api/v1/sandbox/dates          # 获取可用日期列表
GET /api/v1/sandbox/map/skeleton   # 获取地图骨架数据（站点、杆塔）
GET /api/v1/sandbox/map/summary    # 获取地图汇总数据（统计信息）
```

## 原因

### 1. 数据隔离

| 层级 | 访问方式 | 数据内容 |
|------|----------|----------|
| raw_events | Ingestion API（写入） | 上游原始数据 |
| canonical_entities | Normalizer 处理 | 归一化实体数据 |
| Sandbox DTO | Sandbox API（读取） | 下游专用视图 |

- 下游应用看不到 DCP 原始字段
- 字段名和结构贴合下游业务语义
- 未来上游格式变化不影响下游接口

### 2. 支持多下游应用

```
                    ┌─────────────────┐
                    │ canonical_      │
                    │  entities       │
                    └────────┬────────┘
                             │
         ┌───────────────────┼───────────────────┐
         │                   │                   │
         ▼                   ▼                   ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│  Sandbox API    │ │  未来：Report   │ │  未来：Alert    │
│  （数字沙盘）    │ │     API         │ │     API         │
│                 │ │                 │ │                 │
│ /map/skeleton   │ │ /reports/       │ │ /alerts/        │
│ /map/summary    │ │   monthly       │ │   active        │
└─────────────────┘ └─────────────────┘ └─────────────────┘
```

- 每个下游应用可以有独立的 API 接口
- 接口语义贴合应用场景
- 不互相影响

### 3. 安全与治理

- 下游应用只能访问授权的数据
- 原始数据不暴露
- 便于审计和监控

## 影响

### 正面影响

- 下游应用与上游系统完全解耦
- 数据安全：原始字段不暴露
- 接口语义贴合业务场景
- 支持未来扩展新下游应用

### 负面影响

- 需要为每个下游应用维护专用接口
- 增加一层 DTO 转换
- 接口变更需要协调下游应用

## 替代方案

### 方案 A：下游应用直接查询 canonical_entities（已拒绝）

下游应用直接通过 REST API 查询 canonical_entities 表。

**拒绝原因**：
- 暴露内部数据模型，破坏封装
- 下游应用需要了解 entity_type、entity_key 等内部概念
- 无法做字段级别的访问控制
- 接口语义不贴合业务场景

### 方案 B：统一通用查询接口（已拒绝）

提供一个通用的 `/api/v1/query` 接口，下游应用通过参数指定查询条件。

**拒绝原因**：
- 接口过于通用，无法提供贴合业务的语义
- 难以做细粒度的数据隔离
- 下游应用需要了解内部数据结构

## 备注

- Sandbox API 目前为数字沙盘专用，未来可扩展为其他下游应用
- API 路径设计：`/api/v1/sandbox/*` 表示下游应用接口
- 未来版本可考虑添加 API Key 认证，限制下游应用访问范围
- DTO 结构应与下游应用团队协商确定
