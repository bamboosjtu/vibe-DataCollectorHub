# ADR-003: WebSocket单轮询广播模式

## 状态

- 状态: 已接受
- 日期: 2026-03-22

## 背景

MVP阶段需要实现WebSocket准实时推送，但缺乏事件驱动基础设施。

## 决策

**采用"单轮询 + 广播"模式实现WebSocket准实时推送。**

```
架构:
┌─────────────┐
│  单轮询任务  │◄── 每5秒查询一次数据库
└──────┬──────┘
       │ 广播新数据
       ▼
┌───────────────────────────────────┐
│  所有连接的WebSocket客户端          │
│  [Client1, Client2, ..., ClientN]  │
└───────────────────────────────────┘
```

## 原因

### 1. 避免数据库压力

- 问题：100个客户端 → 100个轮询 → SQLite直接炸
- 解决：1个轮询任务，广播给所有客户端

### 2. MVP阶段足够

- 延迟1-5秒可接受
- 实现简单，无需消息队列

### 3. 资源可控

- 固定1个后台任务
- 不受客户端数量影响

## 后果

### 正面

- 数据库压力可控
- 支持大量并发连接
- 实现简单

### 负面

- 延迟高于事件驱动模式（1-5秒 vs 实时）
- 需要轮询，有一定资源消耗

## 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 事件驱动 | 实时性最好 | 需要消息队列，MVP不采用 |
| 多轮询（当前避免） | 实现简单 | 数据库压力随客户端增加 |
| 单轮询广播（采用） | 资源可控 | 准实时，延迟1-5秒 |

## 后续迭代

未来可升级为事件驱动：
- 引入Event bus / message queue
- 实现pub-sub模型
- 支持真正的实时推送
