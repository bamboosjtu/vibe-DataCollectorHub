# ADR-005: 三层数据架构

## 状态

- 状态: 已接受
- 日期: 2026-03-22

## 背景

需要设计数据存储架构，支持从原始采集到分析使用的全链路。

## 决策

**采用三层数据架构：raw → normalized → feature**

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 3.1: raw_data                                        │
│  - 原始采集数据，保留完整性                                  │
│  - 支持数据溯源                                             │
├─────────────────────────────────────────────────────────────┤
│  Layer 3.2: normalized_data (MVP核心)                       │
│  - 轻量级规范化，提取关键字段                                │
│  - 支持去重和基础查询                                       │
├─────────────────────────────────────────────────────────────┤
│  Layer 3.3: feature_data (未来扩展)                         │
│  - 特征工程数据                                             │
│  - 用于机器学习、关系图谱等高级分析                          │
└─────────────────────────────────────────────────────────────┘
```

## 原因

### 1. 渐进式数据处理

- 原始层保留完整性（永不丢失）
- 规范化层提取关键字段（去重、索引）
- 特征层高级加工（ML、图谱）

### 2. 下游需求不确定

- 不同分析工具对数据要求不同
- normalized_data保留原始状态（payload字段）
- 不做严格schema限定

### 3. 职责分离

| 层级 | 职责 | 数据加工方 |
|------|------|-----------|
| raw | 原始存储 | 采集系统 |
| normalized | 轻量规范化 | 采集系统 |
| feature | 高级特征 | 下游工具 |

## 关键设计

### normalized_data轻量提取

```python
def normalize(self, raw_data: Dict, raw_data_id: int) -> Optional[Dict]:
    return {
        "event_type": "news",           # 事件类型
        "event_source": "微博",          # 事件来源
        "entity": ["公司A"],             # 核心实体（可选）
        "event_timestamp": datetime,     # 事件时间
        "unique_key": "hash(...)",       # 去重键
        "payload": raw_data              # 保留原始状态
    }
```

### 下游不做深度加工

**采集系统不做**：
- 实体关系图谱构建
- 深度数据清洗
- 复杂特征工程

**下游工具做**：
- 根据需求定制加工
- 实体关系抽取
- 机器学习特征

## 后果

### 正面

- 系统边界清晰
- 下游灵活性高
- 渐进式演进

### 负面

- 存在数据冗余（raw + normalized）
- 下游需要重复处理

## 数据保留策略

| 表 | 保留时间 | 说明 |
|----|---------|------|
| raw_data | 30天 | 短期保留，用于溯源 |
| normalized_data | 90天 | 中期保留，用于分析 |
| feature_data | 长期 | 长期保留（未来扩展） |
