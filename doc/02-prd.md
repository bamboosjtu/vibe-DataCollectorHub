# Data Collector Hub - 产品需求文档 (PRD)

---

## 1. 目标用户

### 1.1 情报研判人员

| 属性 | 描述 |
|------|------|
| **主要职责** | 监控舆情、分析态势、预警风险 |
| **使用场景** | 配置插件采集多源信息，通过API获取数据输入研判工具 |
| **核心需求** | 实时性、多源整合、告警通知 |
| **技术能力** | 中等，能配置参数、查看监控界面 |

### 1.2 数据分析师

| 属性 | 描述 |
|------|------|
| **主要职责** | 数据清洗、分析建模、报告生成 |
| **使用场景** | 调用REST API获取结构化数据，进行深度分析 |
| **核心需求** | 数据完整性、查询灵活性、历史数据回溯 |
| **技术能力** | 较强，能编写脚本调用API、处理JSON数据 |

## 2. 核心场景

### 2.1 场景一：多源舆情监控

**用户**：情报研判人员

**流程**：
1. 配置微博热搜、知乎热榜、新闻RSS等多个插件
2. 设置5分钟定时采集
3. 通过Web管理界面查看实时数据
4. 异常事件自动触发告警

**价值**：一个平台覆盖多个数据源，降低监控成本

### 2.2 场景二：财经数据分析

**用户**：数据分析师

**流程**：
1. 配置天天基金、东方财富等财经数据源
2. 每小时采集一次估值数据
3. 通过API获取历史数据进行趋势分析
4. 生成投资报告

**价值**：标准化数据格式，减少数据清洗工作量

### 2.3 场景三：实时态势大屏

**用户**：情报研判人员

**流程**：
1. 连接WebSocket实时流
2. 数据实时推送到数字沙盘系统
3. 可视化展示多维度态势

**价值**：准实时数据推送，支撑决策响应

## 3. 数据维度需求

本系统采集的数据服务于宏观、中观、微观三个时间维度的分析需求。

### 3.1 宏观维度 - 长期趋势分析

| 特征 | 说明 |
|------|------|
| **时间尺度** | 季度/年度，历史数据回溯 |
| **数据类型** | 周期性指标、政策文本、行业报告、宏观经济数据 |
| **采集频率** | 每日/每周（低频） |
| **典型插件** | 财经历史数据、政策发布平台、行业白皮书 |

**下游应用**：年度分析报告、政策影响分析

### 3.2 中观维度 - 行业动态监控

| 特征 | 说明 |
|------|------|
| **时间尺度** | 周/月，中期波动跟踪 |
| **数据类型** | 行业新闻、竞品动态、市场情绪、供应链信息 |
| **采集频率** | 每小时/每日（中频） |
| **典型插件** | 行业媒体、社交媒体话题、股票板块数据 |

**下游应用**：竞品监控报告、行业景气度指数

### 3.3 微观维度 - 实时事件响应

| 特征 | 说明 |
|------|------|
| **时间尺度** | 分钟/小时，即时响应 |
| **数据类型** | 突发事件、舆情爆发、价格异动、热点话题 |
| **采集频率** | 每5分钟/实时（高频） |
| **典型插件** | 实时行情、热搜榜单、WebSocket推送 |

**下游应用**：数字沙盘实时态势、突发事件预警

## 4. MVP范围

### 4.1 核心目标

心中一个数据源（插件）并接入调度，能够定时采集并通过 REST API 提供给下游工具。

### 4.2 功能清单

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 插件定时调度 | P0 | APScheduler支持cron表达式 |
| REST API查询 | P0 | 支持分页、过滤、排序 |
| RSS Feed输出 | P1 | 支持标签过滤订阅 |
| WebSocket准实时流 | P1 | 单轮询广播模式 |
| MCP Server | P1 | LLM工具调用接口（query_data/trigger_plugin/list_plugins） |
| 插件失败监控告警 | P0 | 连续失败/失败率告警 |
| Web管理界面 | P0 | Streamlit，仅查看数据和状态 |
| 数据规范化层 | P0 | 提取实体、生成去重键 |

### 4.3 非目标（明确排除）

| 功能 | 说明 | 计划 |
|------|------|------|
| 进程级隔离 | 当前为协程级隔离 | 后续迭代 |
| 多进程/分布式并发 | 单节点协程并发 | 后续迭代 |
| 事件驱动实时流 | 当前为轮询模式 | 后续迭代 |
| 用户管理和权限控制 | 局域网开放访问 | 暂无计划 |
| 复杂数据加工 | 实体关系图谱、深度清洗 | 下游工具实现 |

## 5. SLA / 成功指标

### 5.1 可靠性指标

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 采集成功率 | ≥ 99% | 成功次数/总执行次数（含重试） |
| 调度准时率 | ≥ 99.9% | 实际执行时间-计划执行时间<1秒 |
| 单插件失败影响 | 不影响其他任务 | 故障注入测试 |

### 5.2 性能指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| API P99延迟 | < 200ms | 数据查询接口P99响应时间 |
| WebSocket延迟 | < 1s | MVP阶段准实时轮询延迟 |
| 同时运行插件数 | ≥ 2 | 协程级并发执行能力 |

### 5.3 可观测指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 告警延迟 | < 60s | 插件失败到告警通知时间 |
| 监控数据完整性 | 100% | 任务统计不丢失 |

## 7. 下游工具接入场景

### 7.1 分析报告工具（类似 BettaFish）

**接入方式**：REST API

**使用模式**：
```python
# 获取多维度数据
macro_data = requests.get("/api/data?event_type=finance&limit=1000")
meso_data = requests.get("/api/data?tag=news&limit=100")
micro_data = requests.get("/api/data?plugin_id=weibo_hot&limit=50")

# 综合分析生成报告
generate_report(macro_data, meso_data, micro_data)
```

### 7.2 数字沙盘工具（类似 situation-monitor）

**接入方式**：WebSocket

**使用模式**：
```javascript
const ws = new WebSocket('ws://localhost:8000/ws/stream?plugins=weibo_hot');
ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    updateDashboard(data);  // 实时更新沙盘
};
```

### 7.3 自动化流程

**接入方式**：RSS Feed

**使用模式**：
```bash
curl http://localhost:8000/feed/rss?tag=urgent
# 配合 n8n / Huginn 实现自动化告警
```

### 7.4 LLM/AI Agent 工具调用

**接入方式**：MCP Server

**使用模式**：
```
本系统暴露 MCP 能力端点：http://<服务器>:8000/mcp

LLM 对话示例：
  用户: "帮我查看最近采集的微博热搜"
  LLM: [调用 list_plugins] → [调用 query_data, plugin_id="weibo_hot", limit=10]
  LLM: "最近10条微博热搜如下：..."
```

**客户端接入说明**：
- 具体 MCP 客户端配置方式以所用客户端实现为准（Claude Desktop、Cursor等各有不同）
- 本系统提供标准 MCP 能力端点 `/mcp`，客户端按协议对接即可
- 详见 [05-api-spec.md §6 MCP Server](./05-api-spec.md#6-mcp-server-接口) 和 [ADR-006](./decisions/ADR-006-mcp-server-support.md)

**适用场景**：
- AI 数据分析助手：自然语言查询数据
- 自动化 Agent：根据上下文触发采集
- 智能工作流：LLM 决策 + 数据采集

## 8. 约束与假设

### 8.1 技术约束

- **部署环境**：单机运行，局域网访问
- **存储限制**：SQLite单文件，建议<1GB
- **并发限制**：协程级，默认≤2任务同时执行
- **网络限制**：插件自行处理代理、认证

### 8.2 业务假设

- 用户具备基础技术能力（能配置JSON、看懂日志）
- 下游工具自行处理数据清洗、分析、可视化
- 数据源合法性由用户保证

## 9. 非功能性约束总结

本节汇总系统层面的硬性约束，影响设计决策和技术选型。

### 9.1 部署约束

| 约束项 | 说明 | 影响 |
|--------|------|------|
| **单机部署** | 不支持多节点、分布式 | 无HA、无故障转移 |
| **局域网访问** | 设计为内网使用，无公网安全加固 | 需自行处理网络隔离 |
| **无认证** | 不实现用户管理和权限控制 | 所有内网用户均可访问 |

### 9.2 存储约束

| 约束项 | 说明 | 影响 |
|--------|------|------|
| **SQLite单文件** | 零配置嵌入式数据库 | 不支持高并发写入、单文件大小受限 |
| **存储上限** | 建议<1GB | 需定期清理或归档历史数据 |
| **无分布式存储** | 不支持分片、复制 | 单点故障风险 |

### 9.3 并发约束

| 约束项 | 说明 | 影响 |
|--------|------|------|
| **协程级并发** | asyncio而非多进程 | 插件卡死可能影响主服务 |
| **默认并发数** | ≤2任务同时执行 | 插件多时需要排队 |
| **单线程调度** | APScheduler单实例 | 无分布式任务调度 |

### 9.4 性能约束

| 约束项 | 说明 | 影响 |
|--------|------|------|
| **API P99** | <200ms | 大数据量查询可能超时 |
| **WebSocket延迟** | <1s（轮询模式） | 非真正实时，存在延迟 |
| **采集频率** | 最低5分钟间隔 | 更高频率需外部触发 |

### 9.5 扩展性约束

| 约束项 | 说明 | 影响 |
|--------|------|------|
| **无插件依赖** | 插件必须完全独立 | 无法复用其他插件逻辑 |
| **无插件市场** | 手动安装插件文件 | 无版本管理、自动更新 |
| **无水平扩展** | 不支持增加节点扩容 | 性能瓶颈在单机资源 |
